# 1. What is Docker Compose

Docker Compose is a tool for managing multiple Docker containers together.

Instead of writing long docker commands:

```bash
docker run ...
docker run ...
docker run ...
```

You define everything inside:

```text
docker-compose.yml
```

Then run:

```bash
docker compose up
```

---

# 2. Why Docker Compose Exists

Real applications usually need multiple services:

| Service       | Example  |
| ------------- | -------- |
| Backend       | Node.js  |
| Database      | MySQL    |
| Cache         | Redis    |
| Reverse Proxy | Nginx    |
| Queue         | RabbitMQ |

Managing them manually becomes difficult.

Compose solves this.

# 8. Services Explained

Each service = one container.

Example:

```yaml
services:
  backend:
    build: .

  mysql:
    image: mysql:8
```

Creates:

- backend container
- mysql container

---

# 9. Networks in Compose

Compose automatically creates internal network.

Containers communicate using service names.

Example:

```env
DB_HOST=mysql
```

NOT:

```env
DB_HOST=localhost
```

---

# Why?

Inside containers:

```text
localhost = current container
```

# 14. MySQL + Node.js Example

## docker-compose.yml

```yaml
services:
  app:
    build: .
    ports:
      - "8000:8000"
    env_file:
      - .env
    depends_on:
      - mysql

  mysql:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: admin
      MYSQL_DATABASE: test_db
    ports:
      - "3306:3306"
    volumes:
      - mysql-data:/var/lib/mysql

volumes:
  mysql-data:
```

---

# Important Change

Inside app .env:

```env
DB_HOST=mysql
```

NOT:

```env
DB_HOST=localhost
```

Because Compose networking uses service names.

| Policy         | Meaning           |
| -------------- | ----------------- |
| no             | default           |
| always         | always restart    |
| on-failure     | restart on crash  |

# 24. Production Best Practices

## Recommended

- use named volumes
- use health checks
- separate prod/dev configs
- avoid bind mounts in production
- pin image versions
- use restart policies
- use multi-stage builds

# 25. Multi-Stage Docker + Compose

## Dockerfile

```Dockerfile
FROM node:20 AS builder

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .
RUN npm run build

FROM node:20-alpine

WORKDIR /app

COPY --from=builder /app/dist ./dist
COPY package*.json ./

RUN npm ci --only=production

CMD ["node", "dist/index.js"]
```

# 28. Common Mistakes

## Using localhost Between Containers

❌ Wrong:

```env
DB_HOST=localhost
```

✅ Correct:

```env
DB_HOST=mysql
```

---

## Forgetting Volumes

Without volume:

- database data lost after deletion

---

## Using Development Image in Production

Avoid:

```Dockerfile
npm install
npm start
```

Use:

```Dockerfile
npm ci --only=production
node dist/index.js
```

# 29. Real Production Example

## docker-compose.prod.yml

```yaml
services:
  app:
    build: .
    container_name: backend
    restart: unless-stopped
    ports:
      - "8000:8000"
    env_file:
      - .env
    depends_on:
      - mysql

  mysql:
    image: mysql:8
    container_name: mysql
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: admin
      MYSQL_DATABASE: test_db
    volumes:
      - mysql-data:/var/lib/mysql

volumes:
  mysql-data:
```

---

# Start Production

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

---

# Stop Production

```bash
docker compose -f docker-compose.prod.yml down
```

# Most Important Concepts to Remember

| Concept      | Important Rule                       |
| ------------ | ------------------------------------ |
| localhost    | current container only               |
| service name | container-to-container communication |
| volume       | persistent storage                   |
| bind mount   | live sync                            |
| compose      | manages multiple containers          |
| depends_on   | startup order only                   |
| env_file     | loads environment variables          |
